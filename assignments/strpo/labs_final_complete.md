Лабораторные работы выполняются параллельно с заданиями по дисциплине **«Структуры данных»**.  
Рекомендуется использовать одну и ту же кодовую базу, то есть можно сразу приходить и сдавать лабы по обоим предметам.

В связи с тем, что в курсе по C++ предусмотрено четыре лабораторные работы, лабораторные работы **№1, №2, №3 и №5** по дисциплине *«Современные технологии разработки ПО»*

Использование **GitLab** допускается, однако все вопросы, связанные с отличиями интерфейса и инструментария, обучающийся обязан решать самостоятельно.

Выполнение всех лабораторных работ, помеченных как *необязательные*, до конца семестра позволяет получить автомат по дисциплине «Современные технологии разработки программного обеспечения».

## Лабораторная работа №3  
### Система сборки CMake и работа с библиотеками

### Цель работы  
Освоить современный процесс сборки C++-проекта с использованием CMake и понять основные подходы к линковке библиотек.

Лабораторная работа состоит из трёх этапов.

---

### Этап 1. Базовая сборка проекта

1. Создать файл `CMakeLists.txt` для сборки проекта.
2. Сборка проекта должна выполняться **исключительно** с использованием утилиты `cmake`.
3. Сборка должна выполняться в отдельном каталоге **`build`**.

---

### Этап 2. Статическая библиотека внутри проекта

1. Вынести часть функциональности проекта в статическую библиотеку.
2. Библиотека должна собираться в том же `CMakeLists.txt`, что и основной проект.
3. Для сборки использовать команды:
   - `add_library`;
   - `target_link_libraries`.
4. Заголовочные файлы библиотеки должны подключаться через `target_include_directories`.
5. Использование `file(GLOB ...)` **запрещено**.  
   Все файлы должны быть перечислены явно.

---

### Этап 3. Подключение «внешней» библиотеки через `find_library`

### Цель этапа  
Продемонстрировать подключение библиотеки, поставляемой в виде бинарного артефакта, без доступа к её исходному коду.

### Требования

1. Статическая библиотека должна располагаться в отдельном каталоге и иметь **собственный** `CMakeLists.txt`.
2. Библиотека должна собираться **отдельно** от основного проекта.
3. Основной проект:
   - не имеет доступа к исходным файлам библиотеки;
   - использует только заголовочные файлы и собранный бинарный файл библиотеки.
4. В основном `CMakeLists.txt` **запрещено**:
   - использовать `add_subdirectory` для подключения библиотеки;
   - добавлять исходные файлы библиотеки в сборку.
5. Разрешается использовать только:
   - `target_include_directories`;
   - `find_library(PATHS … NO_DEFAULT_PATH)`;
   - `target_link_libraries`.
6. Поиск библиотеки должен выполняться **строго в заранее заданном каталоге**, без использования системных путей.
7. В случае, если библиотека не найдена, сборка должна завершаться с диагностическим сообщением об ошибке.

---

Этапы 2 и 3 могут быть:

- выполнены в отдельных ветках;

 - либо объединены в одном проекте с демонстрацией обоих подходов (т.е. часть функционала программы должна быть вынесена в соответствии с этапом 2, а другая часть - с этапом 3, собираться все за один проход без переключения веток).

---

> Для автомата надо будет еще продемонстрировать работу с динамическими библиотеками, это необязательная часть данной лабораторной работы, будет описана позднее



---

### Этап 4. Динамические библиотеки *(необязательный, для автомата)*

### Цель этапа  
Познакомиться с динамическими библиотеками и способами их использования:

- неявное связывание (ОС все делает за вас при запуске программы);
- явное связывание (руками во время выполнения программы).

---

## Общие требования

1. Необходимо реализовать **две динамические библиотеки**:

   - первая — используется через **неявное связывание**;
   - вторая — загружается через **явное связывание** во время выполнения.

2. Библиотеки должны реализовывать **разные функции** (можно показывать обе на одном исполняемом файле).

---

## Часть 1. Неявное связывание

Требования:

- создать динамическую библиотеку через `add_library(... SHARED ...)`;
- подключить её через `target_link_libraries`;
- реализовать **любую функцию на выбор студента**;
- вызвать её из основного кода.

---

## Часть 2. Явное связывание

Требования:

1. Библиотека должна:
   - находиться в отдельном каталоге;
   - собираться отдельным `CMakeLists.txt`;
   - не подключаться через `target_link_libraries`.

2. Библиотека должна реализовывать функцию:

```
extern "C" void cat_print(const char* text);
```

Функция должна выводить форматированный текст с ASCII артом, как в одной из лабораторных прошлого семестра.

---

## Подсказки по реализации

### Unix (Linux / macOS)

```
#include <dlfcn.h>
```

Основные функции:

- `dlopen`
- `dlsym`
- `dlclose`

Пример:

```
void* handle = dlopen("./libmylib.so", RTLD_LAZY);

using func_t = void(*)(const char*);
func_t func = (func_t)dlsym(handle, "process");

func("Hello");

dlclose(handle);
```

Линковка (может не требоваться на mac os):

```
target_link_libraries(main PRIVATE dl)
```

---

### Windows

```
#include <windows.h>
```

Основные функции:

- `LoadLibrary`
- `GetProcAddress`
- `FreeLibrary`

Пример:

```
HMODULE lib = LoadLibrary("mylib.dll");

using func_t = void(*)(const char*);
func_t func = (func_t)GetProcAddress(lib, "process");

func("Hello");

FreeLibrary(lib);
```

## Важные замечания

- Путь к библиотеке должен быть корректным (относительный или абсолютный).
- Имя функции в dlsym / GetProcAddress должно точно совпадать.
- Нельзя использовать C++-перегрузки для экспортируемой функции.

## Лабораторная работа №4 *(необязательная)*  
### Отладка и анализ ошибок в C++

### Цель работы  
Продемонстрировать понимание типовых ошибок в C++ и способов их обнаружения с использованием инструментов динамического и статического анализа.

### Задание

Подготовить отчёт, в котором представлены **минимальные примеры кода**, приводящие к следующим проблемам:

1. Segmentation fault  
2. Утечка памяти  
3. Ошибка линковки  
4. Undefined behaviour  

### Обязательные требования

- **Статический анализ кода должен быть продемонстрирован обязательно**  
  (clang-tidy или PVS-Studio).
- Для остальных видов анализа допускается выбор инструментов.

### Допустимые инструменты

- флаги компиляции (`-Wall`, `-Wextra`, и т.п.);
- AddressSanitizer / UndefinedBehaviorSanitizer;
- Valgrind (Linux);
- CRT Debug Heap (CRTDBG) при использовании MSVC;
- отладчик (gdb / Visual Studio).

Для каждого случая необходимо:
- привести код, вызывающий ошибку;
- объяснить причину возникновения;
- продемонстрировать обнаружение проблемы выбранными инструментами.

## Лабораторная работа №5

### Цель работы  
Освоить базовые принципы автоматической сборки и тестирования проекта.

### Задание

1. Реализовать несколько тестов для проверки корректности работы проекта.
2. Для тестирования использовать **только инфраструктуру CMake и утилиту `ctest`**.
3. Все тесты должны:
   - собираться через CMake;

## Лабораторная работа №6  
### Трекинг задач, Kanban и сопровождение курсовой работы

### Цель работы  
Освоить базовые принципы управления разработкой с использованием трекера задач и Kanban-доски, а также интегрировать их в процесс выполнения курсовой работы.

---

### Важное условие

Данная лабораторная работа **обязательна для зачёта курсовой работы по C++**.

> В случае сдачи курсовой работы без выполнения требований данной лабораторной работы  
> студент считается **не выполнившим курс лабораторных работ по дисциплине «Современные технологии разработки программного обеспечения»**  
> и направляется на дополнительную сессию.

---

### Общая идея

В рамках данной лабораторной работы необходимо:

- вести разработку курсового проекта **через трекер задач**;
- фиксировать **планирование, ход выполнения и принятые решения**;
- структурировать работу с использованием **Kanban-доски**.

Цель - не создать дополнительную бюрократию, а:

- научиться работать с инструментами, используемыми в промышленной разработке;
- разбить работу на управляемые этапы;
- сделать процесс разработки **последовательным и осознанным**.

> В реальных проектах подобные инструменты используются в командной работе.  
> В рамках данного курса работа выполняется индивидуально, к сожалению, групповую деятельность сложно встроить в текущий список лабораторных.

---

### Выбор инструмента

Рекомендуется использовать один из двух трекеров задач:

- YouTrack
- GitHub Projects

---

### Организация работы

#### 1. Структура задач

1. Разбить курсовую работу на **5–10 задач (features)**.
2. Каждая задача должна:
   - иметь отдельную карточку в трекере;
   - соответствовать **одной логически завершённой функциональности**;
   - выполняться в **отдельной ветке**.

---

#### 2. Связь задач с Git

Каждая задача должна быть связана с:

- отдельной веткой (`feature/*`);
- коммитами;
- Pull Request.

---

#### 3. Требования для YouTrack

- использовать идентификатор задачи (например, `ABC-123`);
- указывать его:
  - в названии ветки (`feature/ABC-123_feature`)
  - в коммитах (`[ABC-123] ...`)

---

#### 4. Требования для GitHub Projects

- каждая задача — Issue;
- использовать номер Issue:
  - в ветке (`feature/#2_feature`)
  - в коммитах (`[#2] ...`);
- PR должен быть [связан с Issue](https://docs.github.com/en/issues/tracking-your-work-with-issues/using-issues/linking-a-pull-request-to-an-issue) и закрывать его.

Я подготовил [небольшой пример](https://github.com/users/Stasychbr/projects/3), как это должно выглядеть.

---

### Kanban-доска

Выбираем именно этот вариант организации, так как он самый простой и понятный. 

На доске необходимо сделать минимум 3 столбца:

- Backlog  
- In Progress  
- Done  

Можно также добавить `Testing`, если есть желание.

---

### Работа с репозиторием

Рекомендуется сделать `fork` оригинального репозитория с курсовой работой и работать в нем как в своем, чтобы не копировать код вручную.

---

### Pull Request

- Необходимо делать PR из feature веток в dev и из dev в main;
- В main ветку не нужно мерджить каждый коммит, по моим оценкам, коммита в main должно быть 2-3 (сам бот и тесты, может +1 еще для чего-то).

---

### Тестирование и CI

- минимум 3 теста;
- интеграция с GitHub Actions.

## Лабораторная работа №7 *(необязательная)*  
### Контейнеризация с использованием Docker

### Цель работы  
Познакомиться с концепцией контейнеризации и научиться запускать приложение в изолированном окружении с использованием Docker.

---

### Теоретическая часть

Docker — инструмент для запуска приложений в изолированном окружении.  
Контейнер использует общее ядро ОС, но изолирует зависимости и файловую систему. Это не полноценная виртуальная машина, она на умеет эмулировать другое железо. Подробнее можно прочитать [на хабре](https://habr.com/ru/companies/netologyru/articles/967546/).

---

### Практическая часть

#### 1. Установка

```
docker --version
docker run hello-world
```

---

#### 2. Запуск Ubuntu

```
docker run -it ubuntu bash
```

---

#### 3. Сборка своего образа

Создать Dockerfile:

```
FROM ubuntu:22.04

RUN apt-get update && apt-get install -y g++

WORKDIR /app

COPY . .

RUN g++ main.cpp -o main

CMD ["./main"]
```

В качестве программы предлагается форматировать сообщение пользователя с помощью ASCII арта, но можно выбрать что-то самостоятельно.

Сборка и запуск:

```
docker build -t my_app .
docker run my_app
```
---

При сдаче работы надо будет рассказать, что означают команды в Dockerfile, объяснить разницу между image и container, и, возможно, ответить еще на какие-то вопросы.
